# Controllo di Gestione v2.0 — Schema Changes

**Versione target:** CG v2.0
**Data:** 2026-04-10
**Stato:** Approvato (2026-04-10) — vedi `v2.0-decisioni.md` per le scelte finali di Marco

Questo documento descrive tutte le modifiche a schema DB previste per la v2.0. Ogni modifica e' NULL-safe e rollback-safe: se la v2.0 va male, un revert del codice lascia un DB ancora leggibile dal codice vecchio.

**NB — nomi dei campi:** la bozza iniziale di questo doc usava `data_scadenza_override` come nome del campo per la modifica della scadenza. La decisione finale (A1 in `v2.0-decisioni.md`) e' di avere invece DUE campi separati: `data_prevista_pagamento` (quando si prevede di pagare) e `data_effettiva_pagamento` (quando si paga davvero), preservando `data_scadenza` come dato analitico XML intoccabile. Le sezioni sotto sono aggiornate a questi nomi. Dove vedi "override" legato a data scadenza e' un residuo della bozza — leggi `v2.0-decisioni.md` §A1 per la versione definitiva.

## 1. Nuove colonne

### 1.1 `fe_fatture.rateizzata_in_spesa_fissa_id` (mig 055)

```sql
ALTER TABLE fe_fatture ADD COLUMN rateizzata_in_spesa_fissa_id INTEGER;
CREATE INDEX idx_fatture_rateizzata ON fe_fatture(rateizzata_in_spesa_fissa_id)
  WHERE rateizzata_in_spesa_fissa_id IS NOT NULL;
```

**Uso:** quando una fattura viene assorbita da un piano di rateizzazione, questo campo punta alla `cg_spese_fisse.id` del piano. La query dello Scadenzario la esclude di default.

**Semantica:**
- `NULL` → fattura normale, come sempre
- `<integer>` → fattura rateizzata, NON mostrare in Scadenzario (salvo filtro "Includi rateizzate")

**Rollback:** il codice vecchio ignora la colonna. Nessun impatto.

---

### 1.2 `fe_fatture.data_scadenza_override` (mig 056)

```sql
ALTER TABLE fe_fatture ADD COLUMN data_scadenza_override TEXT;
```

**Uso:** permette all'utente di sovrascrivere la scadenza calcolata dal parsing XML o dal default fornitore. La query dello Scadenzario usera' `COALESCE(data_scadenza_override, data_scadenza)`.

**Perche' non modificare direttamente `data_scadenza`?** Per preservare il valore originale estratto dall'XML. In futuro potremmo voler mostrare "scadenza originale / scadenza modificata" per audit.

**Semantica:**
- `NULL` → usa `data_scadenza` (il valore calcolato)
- `<date>` → usa questa data (override manuale)

---

### 1.3 `fe_fatture.iban_beneficiario` (mig 056)

```sql
ALTER TABLE fe_fatture ADD COLUMN iban_beneficiario TEXT;
```

**Uso:** IBAN specifico della fattura, che puo' differire dall'IBAN default del fornitore (`suppliers.iban` aggiunto in mig 054). Verra' popolato:
- **Futuro v2.2:** automaticamente dal parsing XML (`DettaglioPagamento/IBAN`)
- **v2.0:** a mano dall'utente se serve

**Fallback chain per la stampa batch:**
1. `fe_fatture.iban_beneficiario` (se valorizzato)
2. `suppliers.iban` (default fornitore)
3. `cg_spese_fisse.iban` (per le rate)
4. Stringa vuota

---

### 1.4 `fe_fatture.modalita_pagamento_override` (mig 056)

```sql
ALTER TABLE fe_fatture ADD COLUMN modalita_pagamento_override TEXT;
```

**Uso:** analogo a `data_scadenza_override` ma per la modalita' di pagamento. Anche qui `COALESCE(modalita_pagamento_override, modalita_pagamento)`.

## 2. Backfill

### 2.1 Rateizzazioni esistenti (mig 057 — CRITICO)

**Obiettivo:** identificare le fatture che sono state rateizzate in passato (quando il flag non esisteva) e flaggarle retroattivamente.

**Strategia di matching:**

Per ogni `cg_spese_fisse` di tipo RATEIZZAZIONE, cerchiamo la fattura originale combinando:

1. **Fornitore**: `cg_spese_fisse.fornitore_piva == fe_fatture.fornitore_piva`
2. **Importo**: `ABS(cg_spese_fisse.importo_originale - fe_fatture.totale_fattura) < 0.01`
3. **Data**: `fe_fatture.data_fattura` entro ±60 giorni da `cg_spese_fisse.created_at` o da `cg_spese_fisse.data_inizio`
4. **Stato**: `fe_fatture.rateizzata_in_spesa_fissa_id IS NULL` (non gia' flaggata)

**Modalita' dry-run OBBLIGATORIA:**

La migrazione 057 **non esegue** l'UPDATE direttamente. Stampa un report dei match proposti (CSV) in `app/data/backfill_057_dryrun.csv` con le colonne:

```
sf_id, sf_titolo, sf_importo, sf_fornitore, candidato_fatture_id, candidato_numero, candidato_data, candidato_totale, match_score
```

Marco rivede il CSV, elimina le righe sbagliate, lo salva come `backfill_057_approved.csv`, e LANCIA uno script `apply_backfill_057.py` che fa l'UPDATE effettivo.

**Perche' cosi' paranoici?** Perche' flaggare una fattura sbagliata significa farla sparire dallo Scadenzario senza preavviso. Un errore silenzioso in cui Marco si accorge del problema settimane dopo.

## 3. Query cambiate

### 3.1 `GET /controllo-gestione/uscite` — riscrittura con JOIN

**Prima (v1.7.1):**

```sql
SELECT
    u.*,
    f.modalita_pagamento AS mp_xml,
    s.modalita_pagamento_default AS mp_default,
    sf.titolo AS sf_titolo,
    sf.tipo AS sf_tipo,
    ...
FROM cg_uscite u
LEFT JOIN fe_fatture f ON u.fattura_id = f.id
LEFT JOIN suppliers s ON u.fornitore_piva = s.partita_iva
LEFT JOIN cg_spese_fisse sf ON u.spesa_fissa_id = sf.id
...
```

I campi "anagrafica" (data_scadenza, totale, fornitore_nome) vengono da `u.*` cioe' da `cg_uscite` — denormalizzati.

**Dopo (v2.0):**

```sql
SELECT
    u.id, u.tipo_uscita, u.stato, u.pagamento_batch_id, u.in_pagamento_at,
    u.pagata_at, u.nota_pagamento, u.fattura_id, u.spesa_fissa_id,
    u.periodo_riferimento, u.note,

    -- Anagrafica: scegliere la fonte giusta in base al tipo
    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN f.fornitore_nome
        WHEN 'SPESA_FISSA' THEN sf.titolo
        ELSE u.fornitore_nome
    END AS fornitore_nome,

    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN f.fornitore_piva
        WHEN 'SPESA_FISSA' THEN sf.fornitore_piva
        ELSE u.fornitore_piva
    END AS fornitore_piva,

    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN f.numero_fattura
        ELSE u.numero_fattura
    END AS numero_fattura,

    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN f.data_fattura
        ELSE u.data_fattura
    END AS data_fattura,

    -- Scadenza: override ha precedenza per le fatture
    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN COALESCE(f.data_scadenza_override, f.data_scadenza, u.data_scadenza)
        WHEN 'SPESA_FISSA' THEN pr.data_scadenza
        ELSE u.data_scadenza
    END AS data_scadenza,

    -- Importo: sempre dalla fonte
    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN f.totale_fattura
        WHEN 'SPESA_FISSA' THEN pr.importo
        ELSE u.totale
    END AS totale,

    u.importo_pagato,

    -- Modalita' e IBAN: override > default > fonte specifica
    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN COALESCE(f.modalita_pagamento_override, f.modalita_pagamento, s.modalita_pagamento_default)
        WHEN 'SPESA_FISSA' THEN NULL
        ELSE NULL
    END AS modalita_pagamento,

    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN COALESCE(f.iban_beneficiario, s.iban)
        WHEN 'SPESA_FISSA' THEN sf.iban
        ELSE NULL
    END AS iban_pagamento,

    -- Batch (come prima)
    pb.titolo AS batch_titolo,
    pb.stato AS batch_stato,

    -- Flag rateizzata
    f.rateizzata_in_spesa_fissa_id

FROM cg_uscite u
LEFT JOIN fe_fatture f ON u.fattura_id = f.id
LEFT JOIN suppliers s ON (
    CASE u.tipo_uscita
        WHEN 'FATTURA' THEN f.fornitore_piva
        ELSE u.fornitore_piva
    END
) = s.partita_iva
LEFT JOIN cg_spese_fisse sf ON u.spesa_fissa_id = sf.id
LEFT JOIN cg_piano_rate pr ON u.spesa_fissa_id = pr.spesa_fissa_id AND u.periodo_riferimento = pr.periodo
LEFT JOIN cg_pagamenti_batch pb ON u.pagamento_batch_id = pb.id

WHERE {where_sql}
  AND (f.rateizzata_in_spesa_fissa_id IS NULL OR :includi_rateizzate = 1)

ORDER BY {order_sql}
```

**Nota:** questa query e' uno **sketch**, va validata con `EXPLAIN QUERY PLAN` prima di metterla in produzione. Potrebbe essere piu' pulita spezzandola in due (una per FATTURA, una per il resto) + UNION ALL. Da decidere nella fase B.

### 3.2 Smart dispatcher modifica scadenza

**Nuovo endpoint logico** (lo vecchio PATCH `/uscite/{id}/scadenza` cambia implementazione):

```python
@router.patch("/uscite/{uscita_id}/scadenza")
def modifica_scadenza_uscita(uscita_id: int, payload: dict, current_user=Depends(get_current_user)):
    nuova_data = payload["data_scadenza"]
    conn = get_fc_db()
    try:
        u = conn.execute("SELECT tipo_uscita, fattura_id, spesa_fissa_id, periodo_riferimento FROM cg_uscite WHERE id = ?", (uscita_id,)).fetchone()
        if not u:
            raise HTTPException(404)

        if u["tipo_uscita"] == "FATTURA" and u["fattura_id"]:
            # Scrive sulla fonte originale
            conn.execute("UPDATE fe_fatture SET data_scadenza_override = ? WHERE id = ?",
                         (nuova_data, u["fattura_id"]))
        elif u["tipo_uscita"] == "SPESA_FISSA" and u["spesa_fissa_id"] and u["periodo_riferimento"]:
            conn.execute("UPDATE cg_piano_rate SET data_scadenza = ? WHERE spesa_fissa_id = ? AND periodo = ?",
                         (nuova_data, u["spesa_fissa_id"], u["periodo_riferimento"]))
        else:
            # STIPENDIO, ALTRO: resta su cg_uscite perche' non hanno altra fonte
            conn.execute("UPDATE cg_uscite SET data_scadenza = ? WHERE id = ?",
                         (nuova_data, uscita_id))
        conn.commit()
        return {"ok": True}
    finally:
        conn.close()
```

## 4. Colonne zombie (da rimuovere in v2.3)

Dopo la v2.0, queste colonne di `cg_uscite` diventano **lette-ma-non-scritte** — il codice le ignora, ma restano in DB per rollback. In v2.3 (dopo rodaggio) verranno DROP:

- `cg_uscite.fornitore_nome` (viene da `fe_fatture` / `cg_spese_fisse`)
- `cg_uscite.fornitore_piva` (stessa cosa)
- `cg_uscite.numero_fattura` (solo per FATTURA → `fe_fatture`)
- `cg_uscite.data_fattura` (solo per FATTURA → `fe_fatture`)
- `cg_uscite.data_scadenza` (per FATTURA → `fe_fatture`, per SPESA_FISSA → `cg_piano_rate`; resta per STIPENDIO/ALTRO)
- `cg_uscite.totale` (stessa cosa)

**NB:** la colonna `cg_uscite.data_scadenza` NON puo' essere droppata del tutto perche' STIPENDIO e ALTRO non hanno una fonte alternativa. Verra' mantenuta ma usata solo per quei tipi.
